# 2026-05-01 — Lead flow upgrade (Notion + Resend + N8N)

End-to-end rebuild of the lead capture, nurture, and follow-up pipeline. Replaces the prior Web3Forms-only flow with a centralized Notion CRM, a Resend 5-email nurture sequence, N8N-driven Cal.com pre-meeting reminders, and Resend inbound forwarding to Katherine's Gmail.

## What changed

### Forms → first-party API routes (Phase 1)

`/inscribete` and `/viajes-privados` now POST to first-party API routes (`/api/leads/form`, `/api/leads/private`) instead of Web3Forms. Hard cutover — `WEB3FORMS_KEY` removed from both components, zero references remain in `src/`.

### Notion "Sales CRM" as central database (Phase 1)

Lead rows now flow into the existing "Sales CRM" Notion database (id `699c629a2dbc8242a7b28153361201ca`). DB schema patched via API to add columns:
- `Source` (select: `Form` / `PrivateForm` / `Cal`)
- `Personas` (number)
- `Comentarios` (rich_text)
- `Cal Booking Start` (date)
- `Cal Booking UID` (rich_text — used as match key for Cal events)

`Viaje` select extended with `Abril 2027` and `Ambas fechas` (was only `Agosto 2026`).

The existing `Status` property (Notion `status` type) is reused with the existing options Lead/Qualified/Meeting/Proposal/Negotiation/Lost/Closed. New writes:
- Form submission → `Lead`
- Cal booking created → `Meeting`
- Cal booking cancelled → `Lost`
- Meeting ended → `Qualified`

`Cal Booking Start` is stored in **Vietnam local time** (`Asia/Ho_Chi_Minh`, UTC+7) so that Katherine sees the meeting time in her own working timezone when reviewing the CRM.

### Resend nurture sequence (Phase 2)

Setup in Resend:
- Audience `form-leads` (id `ba6e3b37-ba36-48bc-a63d-a83385b3b69a`)
- 5 published templates: `als-welcome`, `als-scarcity`, `als-destino`, `als-curated`, `als-final-cta`
- Custom event `lead.form_submitted` (schema: source string, viaje string, personas number)
- Automation `form-leads · 5-email nurture` (id `019de138-1ced-75dd-bc65-235f02fb7934`): trigger → 5 `send_email` steps with delays `1d / 2d / 3d / 3d`. **Currently enabled.**

Final-CTA template uses a single WhatsApp link (`wa.me/584248455010`) with a prefilled message in Spanish — replaces the dual reserve/agenda call CTA per user feedback.

Sender identity: `Katherine Molinares · Asia Lo Posible <katherine@emails.asialoposible.net>`. Dedicated marketing subdomain — separates deliverability reputation from the root domain. DNS verified in Resend.

### Resend Inbound + Gmail "Send mail as" (Phase 4)

Receiving enabled on `emails.asialoposible.net` in Resend (MX records added by user).

Gmail "Send mail as" configured with SMTP `smtp.resend.com:465` (username `resend`, password = RESEND_API_KEY). Verified working — Katherine can now compose emails from `katherine@emails.asialoposible.net` directly in her Gmail account.

Inbound webhook handler `src/app/(frontend)/api/webhooks/resend-inbound/route.ts` written: receives `email.received` events from Resend, verifies svix signature with `RESEND_WEBHOOK_SECRET`, fetches the email body, and forwards to `katherinemolinares1@gmail.com` via `resend.emails.receiving.forward({ passthrough: true })`. **Sleeping until first deploy** — once live, register the webhook in Resend Dashboard → Webhooks pointing to `https://asialoposible.net/api/webhooks/resend-inbound`.

### N8N Cal.com Lead Pipeline (Phase 3)

Workflow `Cal.com Lead Pipeline` (id `Er3DYB9jfG1BsX9n`) live and **active** at `https://innovalygroup.app.n8n.cloud`. Public webhook URL: `https://innovalygroup.app.n8n.cloud/webhook/cal-leads`. HMAC secret stored in 1Password under `Cal.com - asialoposible` field `n8n_webhook_hmac_secret`.

Cal.com webhook **registered and ping/test verified** (200 OK). Subscribes to `BOOKING_CREATED`, `BOOKING_RESCHEDULED`, `BOOKING_CANCELLED`, `MEETING_ENDED`.

Workflow logic:
- **`BOOKING_CREATED`**: HMAC verify → parse → create Notion row (Source=Cal, Status=Meeting, time stored in Vietnam tz) → fallback to Katherine's Gmail if Notion fails → schedule T-24h reminder (with status check before send) → schedule T-10m reminder (with status check before send).
- **`BOOKING_RESCHEDULED`**: find Notion row by Cal Booking UID, update Cal Booking Start (status preserved as Meeting).
- **`BOOKING_CANCELLED`**: find row by UID, set Status=Lost. The pending Wait nodes for the same booking still fire when their time comes, but the status check immediately before each send sees `Status !== Meeting` and skips the send. This is how stale reminders are prevented even though Wait nodes can't be aborted.
- **`MEETING_ENDED`**: row Status `Meeting → Qualified`.

Reminder email templates (rendered in the `Verify HMAC + Parse` Code node, then passed to HTTP-Resend nodes):
- **T-24h**: subject includes the meeting time in the **attendee's** timezone (`Intl.DateTimeFormat('es', ..., timeZone: attendee.timeZone)`). Body shows the time prominently, the Google Meet link as a button + plain text URL, and Katherine's WhatsApp `+58 424 845 5010` for any questions.
- **T-10m**: subject `Nos vemos en 10 min — link aquí`. Body has the Meet link as a primary button + plain URL + WhatsApp.

Both reminders go to the attendee's email (extracted from `payload.attendees[0].email`), with reply-to `katherine@emails.asialoposible.net`.

### Fallback alerts

If Notion sync fails for any lead source (form, private form, or Cal booking), an alert email goes to `katherinemolinares1@gmail.com` with the full lead data + error trace, so no lead is lost.

- Form routes: `src/lib/fallback-alert.ts` → `sendLeadFallbackAlert()` invoked from API route catch blocks.
- N8N Cal workflow: error branch on the Notion HTTP node fires a Resend send to the same address.
